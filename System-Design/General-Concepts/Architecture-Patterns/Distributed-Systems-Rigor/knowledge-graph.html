Distributed Systems Rigor

"Correctness under failure — Walmart's non-negotiable baseline"

Round 2 Focus
Walmart's distributed systems rigor round tests whether you can design systems that are correct not just when everything works, but specifically when things fail: retries happen, services crash, networks drop, clocks drift.

The Three Pillars

🔁
Idempotency
Same operation, N times = same result. Makes retries safe. Essential for at-least-once delivery.
⚡
Concurrency
Parallel writes, shared state. Prevent lost updates, dirty reads, write skew.
🛡️
Data Integrity
Cross-service consistency. ACID within service. Saga across services. Right consistency model for each use case.

Idempotency — Definition & Why

f(f(x)) = f(x) — applying the same operation multiple times has the same effect as applying it once

The Core Problem

Failure ScenarioWhat Causes Duplicate
Network timeoutClient retries → 2 requests hit server
Consumer crash after processRe-reads same Kafka message after restart
Kafka at-least-once deliveryRebalance during commit → reprocessed
Load balancer retrySame request hits two instances
Client timeout + retryServer processed first request; client sends again

Natural Idempotency (HTTP)

MethodIdempotent?Why
GET✅ YesRead-only, no state change
PUT✅ YesSets state to exact value; N calls = 1 call
DELETE✅ YesAlready-deleted → 204 (no-op)
POST❌ NoCreates new resource each time
PATCH❌ Usually NoIncremental changes (+10) are not idempotent
Critical
POST is how you create orders, payments, reservations. You must engineer idempotency into all POST endpoints.

The Partial Failure Problem

Sequence:
  1. Charge card          → SUCCESS at gateway
  2. Store in DB          → CRASH before commit
  3. Client retries POST  → idempotency key not found (we crashed before storing it)
  4. Charge card again    → DOUBLE CHARGE 💥

Fix: Store idempotency key in SAME TRANSACTION as the DB write.
     If we can't write to idempotency store, we can't guarantee safety → return 503.
💬 Ready-to-Say Answer
"Idempotency is non-negotiable in distributed systems because retries are guaranteed — not optional. My pattern: client generates a UUID before the first attempt, sends it as an Idempotency-Key header, and the server stores the key→result mapping in the same DB transaction as the operation. On duplicate, return the cached result without re-executing. The critical edge case: if idempotency storage fails, fail the whole request. Better to reject than to risk double-processing."
Confidence:

Idempotency Patterns

Four patterns that cover 95% of real scenarios

Pattern 1: Idempotency Key (Most Versatile)

Client: generates UUID once per logical operation (not per retry)
Server: checks idempotency_keys table → if found, return cached result
        if not found → process → store key + result in same TX

Schema:
  CREATE TABLE idempotency_keys (
    idempotency_key VARCHAR(128) PRIMARY KEY,
    operation       VARCHAR(64)  NOT NULL,
    response_body   JSONB        NOT NULL,
    expires_at      TIMESTAMP    NOT NULL  -- 7 days for payments, 24h for orders
  );

Pattern 2: DB UNIQUE Constraint (Simplest)

CREATE TABLE orders (
  id               BIGINT AUTO_INCREMENT PRIMARY KEY,
  client_order_id  VARCHAR(128) UNIQUE NOT NULL,  ← dedup happens here
  amount           DECIMAL(10,2),
  ...
);

INSERT INTO orders (client_order_id, amount)
VALUES ('client-ref-abc', 99.99)
ON CONFLICT (client_order_id) DO NOTHING
RETURNING *;

→ First insert: creates order
→ Duplicate: DB rejects silently, service returns existing order

Pattern 3: Redis SETNX (Cross-instance)

SET event-{id} "processed" NX PX 86400000  -- NX=only if not exists, PX=TTL ms

Result = OK   → first to process, proceed
Result = NIL  → duplicate, skip

⚠ GC Pause Problem: Lock expires while process is paused → two processors!
Fix: Include fencing token in DB write:
  UPDATE inventory SET qty = qty - 1, fence = 42
  WHERE id = 123 AND fence < 42;

Pattern 4: Conditional Update (State Guard)

UPDATE orders
SET    status = 'PAID', paid_at = NOW()
WHERE  id = 5001
AND    status = 'PENDING';  ← guard: only transitions if in expected state

rows_affected = 1 → success
rows_affected = 0 → check if already PAID (idempotent) or in wrong state (error)

Idempotency Anti-Patterns

The mistakes that cause silent data corruption

Anti-PatternWhy It FailsFix
Timestamp-based dedupClock skew: two events at "same" ms slip throughUse event UUID, never timestamp
Trusting "at-most-once"All brokers eventually retry; you WILL get duplicatesDesign for at-least-once always
No TTL on dedup storeMemory exhaustion / storage bloatExplicit TTL + cleanup job
Check-then-insert without TXTwo threads both see "not exists", both insertUse UNIQUE constraint or SETNX (atomic)
Mutable identifiers as keyEmail can change, reference can be reusedUse immutable UUID generated at request time
Idempotency store in separate TXProcess but crash before store → duplicate on retryStore key in SAME transaction as operation
The Silent Killer
Check-then-insert without atomicity: SELECT → no row → INSERT
Two concurrent threads BOTH see "no row" and BOTH insert. Race condition.
Use INSERT ON CONFLICT DO NOTHING or SET NX — atomic by definition.

Race Condition Taxonomy

Name the failure mode before you prescribe the fix

Race TypeWhat HappensExampleFix

Dirty ReadReads uncommitted data from another TXSees balance=0 from a TX that later rolled backRead Committed isolation (min level)
Non-Repeatable ReadSame query returns different results in same TXFirst read: PENDING, second read: CANCELLEDRepeatable Read isolation
Phantom ReadNew rows appear mid-transactionCOUNT(*) returns 5 then 6 in same TXSerializable isolation or range locks

Write Skew — The Hardest to Spot
No single write is wrong. Both transactions look valid in isolation. But the combined effect violates a system invariant. Classic signs: "at least N of X must exist" constraints, balance checks before debit, slot availability before booking.

Optimistic vs Pessimistic Locking

The most common concurrency control decision in senior interviews

Decision Framework

Is conflict frequent? NO → OPTIMISTIC LOCKING (version++) YES → Is operation short (<50ms)? YES → PESSIMISTIC (SELECT FOR UPDATE) NO → QUEUE-BASED SERIALIZATION

Optimistic Locking

-- Read (no lock)
SELECT qty, version FROM inventory WHERE sku = 'ITEM-X';
-- Returns: qty=10, version=5

-- Write (version check)
UPDATE inventory
SET    qty = 9, version = 6
WHERE  sku = 'ITEM-X'
AND    version = 5;        ← fails if someone else updated

rows_affected = 1: success
rows_affected = 0: conflict → retry with fresh read

JPA: @Version annotation handles this automatically
Spring: @Retryable(OptimisticLockException.class)

Pessimistic Locking

BEGIN;
SELECT qty FROM inventory
WHERE  sku = 'ITEM-X'
FOR UPDATE;              ← exclusive row lock; others BLOCK here

-- safe to read/modify
UPDATE inventory SET qty = qty - 1 WHERE sku = 'ITEM-X';
COMMIT;                  ← lock released

Variants:
  FOR SHARE           → shared lock (reads OK, writes block)
  FOR UPDATE NOWAIT   → fail immediately if locked (no waiting)
  FOR UPDATE SKIP LOCKED → skip locked rows (job queue pattern)

Skip Locked — Job Queue Pattern

-- Multiple workers pull orders without collision
SELECT id, order_data FROM pending_orders
WHERE status = 'QUEUED'
LIMIT 10
FOR UPDATE SKIP LOCKED;  ← only grab unlocked rows
DimensionOptimisticPessimistic
Contention levelLow — conflicts rareHigh — conflicts frequent
Failure costRetry (cheap)Wait (blocking)
ThroughputHigh (no lock overhead)Lower (serialized)
Retry storm riskYes (high contention → storm)No (wait in queue)
Best forProfile updates, most CRUDFlash sales, financial debits

Distributed Locks & MVCC

When you need concurrency control across services or want reads to never block

Redis Distributed Lock

// Acquire: SET key requestId NX PX 10000
// Release (Lua — atomic check-and-delete):
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end

Critical: Use your own requestId, not a static string.
Otherwise Process A releases Process B's lock!

⚠ The GC Pause Problem

This Will Come Up
1. Process A acquires lock, TTL = 30s
2. Process A pauses for GC for 35s
3. Lock expires
4. Process B acquires lock, starts processing
5. Process A resumes → BOTH in critical section!

Fix: Fencing token — monotonically increasing counter returned with lock.
DB write includes: WHERE fence_token < current_token → stale writes rejected.

MVCC (Postgres/MySQL)

How it works:
  - Each row has xmin (created by) and xmax (deleted by) transaction IDs
  - SELECT sees rows where xmin ≤ snapshot AND xmax is null/future
  - Writers create new versions; VACUUM cleans old ones
  - Readers NEVER block writers; writers NEVER block readers

Result: Long-running analytics queries don't block OLTP writes.

Isolation levels and MVCC:
  Read Committed   → snapshot at each statement (PG default)
  Repeatable Read  → snapshot at transaction start (MySQL default)
  Serializable     → full serializability (SSI in Postgres)

Deadlock Prevention

Prevention is better than detection — design lock ordering upfront

What Is a Deadlock?

TX A holds lock(Order #100), wants lock(Payment #200)
TX B holds lock(Payment #200), wants lock(Order #100)
→ Circular wait → both block forever

Prevention Strategies

1. Consistent Lock Ordering (Best Practice)

// Always acquire locks in same order (ascending ID)
Long first  = Math.min(orderId, paymentId);
Long second = Math.max(orderId, paymentId);
lock(first); lock(second);  // consistent order = no circular wait

2. Lock Timeouts

SET innodb_lock_wait_timeout = 5;  -- MySQL
SET lock_timeout = '5s';           -- Postgres
-- Throws exception after timeout → retry with backoff

3. Short Transactions

BAD:  BEGIN → external API call (200ms) → DB write → COMMIT  (lock held 200ms)
GOOD: external API call → BEGIN → DB write → COMMIT           (lock held <10ms)

Lock hold time ∝ deadlock probability

4. @Retryable on DeadlockLoserDataAccessException

@Retryable(
  value = DeadlockLoserDataAccessException.class,
  maxAttempts = 3,
  backoff = @Backoff(delay = 100, multiplier = 1.5)
)
@Transactional
public void processTransaction(Long id) { ... }
💬 Ready-to-Say
"Deadlock prevention starts in design: always acquire locks in a consistent global order — e.g., always lock the lower entity ID first. And keep transactions short — the longer you hold a lock, the higher the deadlock probability. Modern databases detect deadlocks automatically and kill one transaction. At the application level, @Retryable on DeadlockLoserDataAccessException with exponential backoff handles the recovery."

CAP Theorem — Practical View

During a network partition, choose:
  CP: Reject reads/writes to maintain consistency (ZooKeeper, etcd, HBase)
  AP: Serve potentially stale data to remain available (Cassandra, DynamoDB)

CA (no partition tolerance): Only on single node — not a distributed system.
Always choose P. The real question: C or A when partition occurs?
SystemTypePartition BehaviorUse When
ZooKeeper / etcdCPStops serving until quorum restoredLeader election, config mgmt
CassandraAPServes stale data; syncs laterSocial feeds, user prefs
DynamoDBAP (default)Eventually consistent readsShopping cart, sessions
PostgreSQL (single)CA*N/A (single node)OLTP, financial data
💬 CAP in One Line
"CAP means: during a network partition, you choose between consistency (refuse to answer) or availability (answer with possibly stale data). I choose the consistency model that matches the business requirement — strong for checkout and payments, eventual for browsing and analytics."

Saga Pattern & 2PC

How to do distributed transactions without the 2PC nightmare

2PC — The Classic (Mostly Avoid)

Phase 1: Coordinator sends PREPARE to all participants
         Each participant: "CAN you commit?" → YES or NO

Phase 2: If ALL YES → COMMIT
         If ANY NO  → ROLLBACK

Problems:
  ❌ Blocking: if coordinator crashes after PREPARE, participants stuck forever
  ❌ Coordinator SPOF
  ❌ Network partition during Phase 2 → some commit, some don't

When to use: Single external system (1 DB + 1 payment gateway), tolerate blocking.
             NOT for microservices with 3+ services.

Saga — The Modern Answer

Each step = local transaction + event
Failure at step N → compensating transactions for steps 1..N-1

Happy Path:
  reserve inventory  → charge payment  → create shipment  → send email

Failure at step 3 (shipping):
  undo step 2: refund payment (idempotent)
  undo step 1: release inventory reservation (idempotent)
  cancel order: notify customer

State persisted in DB — saga survives orchestrator crash and resumes.

Compensating Transaction Rules

StepForwardCompensation
1reserve inventoryrelease reservation
2charge paymentrefund payment
3create shipmentcancel shipment
4send emailsend cancellation email
Compensation Must Be Idempotent
If the saga fails mid-compensation, it will retry the compensation. refund(orderId) must check "already refunded?" before executing.

When Compensation Also Fails

The Hardest Follow-up
Answer: Retry with backoff → after N failures → saga moves to STUCK state → oncall paged → DLQ captures context → manual intervention possible.

Key: "The system is noisily stuck, not silently corrupted."
💬 Ready-to-Say on 2PC vs Saga
"2PC is rarely worth the operational cost in microservices — the coordinator becomes a SPOF and the protocol is blocking. I prefer saga with compensating transactions: each step publishes an event, failure triggers compensation in reverse order. With saga, I trade atomic commits for availability, which is acceptable with good monitoring — stuck sagas alert within 15 minutes, far before customers notice."
Confidence:

What "consistent" means depends on the use case

Read-Your-Writes Implementation

Problem: Order placed, customer redirected to confirmation page.
Page reads from replica. Replica hasn't synced yet → "Order Not Found".

Fix:
  1. After POST /orders → store write_timestamp in session cookie
  2. GET /orders/{id} passes timestamp header
  3. Backend: if replica_lag > 0, route to primary for 5 seconds
  4. After 5 seconds: replica synced, normal routing resumes
The Split Consistency Pattern
Use different consistency models for different operations in the same system:
• Display price: eventual (regional CDN cache, fast, OK to be 30s stale)
• Checkout price: strong (primary DB, accurate at moment of purchase)
This avoids making the primary a bottleneck for catalog browsing.

Scenario: Flash Sale Inventory

50,000 concurrent users, 500 items, zero overselling

"Design inventory deduction for Black Friday flash sale"
⭐⭐ Hard

Name the Failure Modes

1. Lost Update: A reads qty=1, B reads qty=1, both deduct → qty=-1 (oversold)
2. Duplicate Deduction: User retries after timeout → deducts twice
3. Thundering Herd: 50k connections lock same DB row → collapse

Three-Tier Solution

Tier 1: Redis DECR (absorb traffic, fast)
  SET inv:ITEM-X 500
  result = DECR inv:ITEM-X
  if result >= 0: emit RESERVATION_CLAIMED to Kafka
  if result < 0:  return 409 "Sold Out"; INCR inv:ITEM-X

Tier 2: Kafka Consumer → DB (durable)
  BEGIN TX
    INSERT INTO reservations (purchase_req_id, sku)
    ON CONFLICT (purchase_req_id) DO NOTHING   ← idempotency
    UPDATE inventory SET qty = qty - 1 WHERE sku = 'ITEM-X' AND qty > 0
  COMMIT

Tier 3: Reconciliation Job (correctness)
  Every 60s: compare Redis count vs DB count
  If drift > threshold: alert + sync

Trade-offs to State

DecisionRisk
Redis as fast pathRedis restart loses in-flight → Kafka+DB are source of truth
Kafka bufferConsumer lag → reservation confirmed but DB not yet updated → monitor lag
Not using SELECT FOR UPDATEAvoids lock contention at 50k RPS; drift caught by reconciliation

Scenario: Payment Safety

Customer double-clicks Pay. Charge once. Refund once.

"How do you guarantee a customer isn't charged twice on retry?"
⭐ Medium

Charge Idempotency

Frontend: Generate idempotency_key = UUID on page load (not on click)
          Include as header: Idempotency-Key: uuid-abc

Backend:
  BEGIN TX
    SELECT from idempotency_keys WHERE key = 'uuid-abc' FOR UPDATE
    IF found: COMMIT; return cached_response

    result = payment_gateway.charge(amount, card, idempotency_key='uuid-abc')
    INSERT INTO idempotency_keys (key, response, expires_at) VALUES (...)
    INSERT INTO payments (order_id, charge_id, amount)
  COMMIT

⚠ Both inserts in same TX: if charge succeeds but TX fails, next retry
  finds no idempotency key → would charge again.
  Solution: gateway idempotency_key ensures gateway also deduplicates.

Ambiguous Gateway Response (Timeout / 5xx)

DO NOT assume failure. DO NOT retry immediately.

1. Store payment status = 'UNCERTAIN'
2. Async job: query gateway using original reference_id
3. Update based on authoritative answer:
   - CHARGED → fulfill order
   - NOT_CHARGED → mark FAILED, notify customer

This prevents both double-charge (don't retry) and lost order (don't assume failure).

Scenario: Order Saga (4 Services)

Walk through a distributed transaction and its failure recovery

"Payment charged, shipping fails at step 3. What happens?"
⭐⭐ Hard

Happy Path

OrderSaga.start(orderId)
  1. InventoryService.reserve(orderId, items)         → InventoryReserved
  2. PaymentService.charge(orderId, amount)            → PaymentCharged
  3. ShippingService.createShipment(orderId)           → ShipmentCreated
  4. NotificationService.sendConfirmation(orderId)     → OrderConfirmed

Failure at Step 3 (Shipping)

ShipmentFailed event received by saga:
  Compensate step 2: PaymentService.refund(orderId)
    → checks: not yet refunded? → refund → mark REFUNDED
  Compensate step 1: InventoryService.release(orderId)
    → checks: not yet released? → increment qty → mark RELEASED
  OrderService.cancel(orderId)
  → Customer notified: "Sorry, shipping unavailable"

When Compensation Fails

Retry with exponential backoff (3 attempts)
If still failing → saga.status = COMPENSATION_FAILED
→ Alert fires → oncall paged → DLQ captures context
→ Manual intervention: engineer processes refund directly

Key point: "Noisily stuck, not silently corrupt"

Saga State Persistence

CREATE TABLE saga_state (
  saga_id     VARCHAR(128) PRIMARY KEY,
  current_step VARCHAR(64),
  status       VARCHAR(32),  -- IN_PROGRESS, COMPLETED, COMPENSATING, STUCK
  payload      JSONB,
  last_updated TIMESTAMP
);

Monitoring: SELECT * FROM saga_state
            WHERE status IN ('IN_PROGRESS','COMPENSATING')
            AND last_updated < NOW() - INTERVAL 30 MINUTE;
→ Alert threshold: > 0 rows for > 5 minutes

Interview Q&A Quiz

Common follow-ups in Walmart distributed systems interviews

Q: Why store the idempotency key in the same transaction as the operation?
For performance — batching reduces round trips
If we process but crash before storing the key, the next retry has no record and processes again — causing a duplicate
So the key expires at the same time as the operation record
✅ Correct: Partial failures (process but not store) are the core danger. Same TX = atomicity. Either both happen or neither does.
Q: When would you choose pessimistic locking over optimistic?
Always — it's simpler to implement
Never — optimistic is always faster
When contention is high and retry cost/storm risk outweighs the blocking cost of holding a lock
✅ Correct: High contention → optimistic causes retry storms. Flash sale inventory is the canonical example where pessimistic (or queue-based) wins.
Q: What's the GC pause problem with Redis distributed locks?
Redis GC pauses the key store, causing lock timeouts
The JVM process holding the lock pauses for GC, the lock TTL expires, another process acquires the lock, and then both processes are in the critical section simultaneously
GC deletes Redis keys before TTL expires
✅ Correct: Fix with a fencing token — monotonically increasing counter returned with lock. DB writes include WHERE fence < token, so stale writers are rejected.
Q: What's write skew and how do you fix it?
When two writes conflict and one is lost — fixed with optimistic locking
When each transaction reads valid state and makes a valid write, but together they violate a system invariant — fixed with Serializable isolation or SELECT FOR UPDATE on the constraint
When a write is made to a stale read — fixed with Read Committed isolation
✅ Correct: Classic example: 2 doctors both see "2 on call" and both leave. Each write is valid; together they break "at least 1 on call." Fix: SELECT FOR UPDATE on the scheduling table, or Serializable isolation.
Q: Why is 2PC usually the wrong choice for microservices?
It requires XA drivers which don't exist for most languages
The coordinator is a SPOF, the protocol is blocking (participants stuck if coordinator crashes after PREPARE), and it performs poorly at high load
2PC doesn't support ACID transactions across services
✅ Correct: Use Saga instead. Trades atomic commits for availability, which is acceptable with proper compensation and monitoring.
Q: What happens if a saga's compensating transaction also fails?
The saga rolls back to the beginning automatically
The system accepts the inconsistency as a known trade-off
Retry with backoff → saga enters STUCK/COMPENSATION_FAILED state → alert fires → oncall intervenes → DLQ captures context for manual replay
✅ Correct: The key insight: "noisily stuck, not silently corrupt." The system always knows what happened; humans can intervene. This is far better than silent inconsistency.
